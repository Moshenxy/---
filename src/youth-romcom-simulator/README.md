# Youth Rom-Com Simulator

This project is a Tavern Helper frontend that creates an immersive youth romantic comedy simulation experience. It leverages a sophisticated dual-API architecture to separate narrative generation from game state management, allowing for complex, dynamic storytelling.

## Architecture

The simulator is built on a "post-processing" model that uses two distinct AI calls:

1.  **Narrative Generation (Main API)**: The primary API is the standard Tavern/SillyTavern API. It is responsible for generating all the narrative content, including the character's thoughts, dialogue, actions, and diary entries, based on the user's input and a comprehensive context built by the `ContextService`.

2.  **State Analysis & Update (Secondary API)**: After the main API generates a new message, a custom, secondary "Analysis API" is called. This API's sole purpose is to analyze the narrative and the context that produced it, and then output a command to update the game's state variables.

### State Update Workflow

The state update process is triggered automatically after each new message is generated by the main API. Here is the step-by-step workflow, orchestrated by the `StateUpdateService`:

1.  **Event Interception**: A listener intercepts the `GENERATION_ENDED` event, which fires after the main API completes its response.

2.  **Asynchronous Delay**: A brief 100ms delay is introduced. This is a critical step to resolve a race condition, ensuring that by the time the service proceeds, Tavern's internal state (including the latest message content and the context stored in the `DebugService`) has been fully updated and is safe to read.

3.  **Context & Rule Aggregation**: The service gathers all necessary information for the Analysis AI:
    *   **Main AI Context**: It retrieves the full, rich context that was just used by the main AI to generate its narrative. This is fetched from the `DebugService`.
    *   **Update Rules**: It dynamically collects all relevant state update rules from the character's loaded Worldbook, specifically looking for entries containing `[mvu_update]` and `[cot]µĢ░µŹ«ń▓Šń«ŚÕ«ż`.
    *   **Narrative Text**: It reads the final, new message content generated by the main AI.

4.  **Analysis API Call**: A comprehensive prompt is sent to the configured secondary "Analysis API". This prompt includes the main AI's context, the update rules, and the resulting narrative text.

5.  **MVU String Generation**: The Analysis API processes this rich information and returns a single string formatted as an **MVU (Malleable Variable Unit)** command (e.g., `<[HP|ADD|-10]>`). This string represents the calculated state change.

6.  **Message & State Update**:
    *   The returned MVU string is appended to the end of the original message from the main AI.
    *   The message content in the chat is updated with this new, combined string using `setChatMessages`.
    *   The `Mvu.parseMessage` function is called on the updated message, which automatically parses the MVU command and applies the changes to the game's state variables.

## Core Components

-   `src/youth-romcom-simulator/services/StateUpdateService.ts`: The heart of the state management system. It orchestrates the entire post-processing workflow.
-   `src/youth-romcom-simulator/services/ContextService.ts`: Builds the rich context for the main AI, which is also passed to the secondary AI.
-   `src/youth-romcom-simulator/services/DebugService.ts`: A utility service that, among other things, temporarily stores the last used AI context so it can be retrieved by the `StateUpdateService`.
-   `src/youth-romcom-simulator/store/settingsStore.ts`: A Pinia store that manages the Analysis API settings.
