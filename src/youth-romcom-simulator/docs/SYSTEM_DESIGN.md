# 综漫-春物：神谕引擎 v2 - 设计文档

## 摘要

本文档旨在详细阐述“综漫-春物”世界模拟器V2版本的核心架构——**神谕引擎 (The Oracle Engine)**。该引擎是一套基于前端 `transform` 函数构建的全链路、高度自动化的世界模拟系统。其核心设计哲学为：**前端负责定量计算，AI负责定性描述**。引擎的目标是将一个静态的世界观数据库，转化为一个能够自我演化、充满生机与动态的“活”世界，同时将AI的职责从“逻辑决策者”简化为“高保真叙事渲染器”。

---

## I. 核心架构：神谕引擎 (The Oracle Engine)

“神谕引擎”是位于 `变量结构.txt` 文件核心的 `transform` 函数。它不再是一个简单的变量转换器，而是一个自洽的、在每次世界状态变更时都会执行的微型“世界模拟器”。引擎的执行是幂等的、无副作用的，并遵循严格的计算顺序，以确保世界演化的逻辑自洽性。

### 系统流程与数据流

引擎的每一次完整执行都遵循下图所示的精密流程，该流程涵盖了从时间流逝到信息传播的完整模拟：

```mermaid
graph TD
    subgraph 神谕引擎 (transform 函数)
        A[开始] --> B(冷却池清理 & 状态校正);
        B --> C{AP <= 0?};
        C -- 是 --> D(时间推进);
        D --> D1(<b>NPC位置更新</b>);
        D1 --> D2(<b>地点特质激活</b>);
        D2 --> E(状态自动清理);
        E --> F{新的一天?};
        F -- 是 --> G(日结, 命运预言, <b>NPC轨迹生成</b>);
        G --> H(氛围张力计算);
        F -- 否 --> H;
        H --> I[结束, 输出新状态];
        C -- 否 --> H;
    end

    subgraph AI 叙事引擎
        I -- 读取新状态 --> J(AI读取完全计算好的世界状态);
        J --> K{检查偶遇 & 场景特质};
        K --> L[渲染叙事, 提交变量变更];
    end
```

## II. 自动化子系统详解

神谕引擎由多个相互协作的自动化子系统构成，共同驱动着世界的动态演化。

### 2.1 动机驱动的动态日程系统 (Motive-Driven Dynamic Scheduler)

这是引擎的“大脑”，决定了NPC如何“活”起来。

*   **机制**: 摒弃了固定的日程表。每个NPC的行动选项都包含一个**基础权重**和一系列**权重修正器**。
*   **动态计算**: 引擎在每日开始时，会根据每个NPC**当前的内在状态**（如心理健康）、**外部环境**（如天气）、以及**人际关系**（如与主角的亲密度），动态计算出每个行动选项的最终权重。
*   **效果**: NPC的行为变得可解释且充满人格。一个“承压”的雪乃会自动倾向于去图书馆独处，而不是参加社交活动；一个与主角关系升温的结衣，会更频繁地出现在主角的核心活动区。

### 2.2 动态场景特质引擎 (Dynamic Scene Trait Engine)

这是引擎的“画笔”，让世界本身变得有“表情”。

*   **机制**: 为每个地点预设多个带有`触发条件`的“场景特质”（如“静谧的红茶时光”、“女王的审判席”）。
*   **自动激活**: 引擎在每次时间片段更新后，会**自动扫描**所有地点，根据当前的时间、天气、在场角色等条件，计算出哪些特质应被**激活**。
*   **效果**: AI不再需要猜测场景氛围。它只需读取地点`当前激活特质`列表，并忠实地将特质的描述和效果渲染到叙事中。世界场景的氛围变化实现了完全的自动化和逻辑化。

### 2.3 信息传播与流言系统 (Information & Rumor Propagation System)

这是引擎的“神经网络”，模拟了现实世界中至关重要的信息生态。

*   **机制**: 引入“信息包”这一新实体。当一个事件发生时，会创建一个包含“事实”的初始信息包，并标记知情者。
*   **自动传播**: 在每日结算时，引擎会模拟信息在角色关系网络中的传播。传播概率受**信息劲爆程度**和**角色间亲密度**影响。
*   **失真与变异**: 每次传播都有一定概率（失真率）使信息内容发生变异，从而产生“流言”和“误解”。
*   **效果**: NPC的认知不再局限于亲身经历。他们会基于听来的“八卦”做出判断和行动，这为世界自动地、源源不断地创造了无数充满误解和戏剧性的“春物式”剧情。

### 2.4 辅助自动化模块

*   **状态自动清理**: 引擎自动检测并移除所有角色身上已过期的状态（status）。
*   **氛围张力计**: 自动计算当前场景中所有角色间的关系力场，得出一个量化的“张力值”和“氛围”状态（如“尴尬”、“紧张”）。
*   **关系冷却池**: 自动管理角色间的社交动作冷却时间，防止AI在短时间内重复单调的互动。

## III. AI职责白皮书

随着神谕引擎的全面实装，AI的职责被前所未有地精简和聚焦：

*   **核心定位**: **高保真叙事渲染器**。
*   **禁止事项**: 严禁AI进行任何形式的逻辑判断、概率计算或状态推演。所有这些工作都已由神谕引擎完成。
*   **核心任务**:
    1.  **忠实读取**: 在每一轮开始时，完整、忠实地读取由引擎计算出的最终世界状态。
    2.  **情景渲染**: 将所有自动计算出的结果（NPC的偶遇、地点的激活特质、场景的氛围）作为不可辩驳的事实，用高质量的文学语言渲染成生动的叙事。
    3.  **响应玩家**: 在渲染好的情景中，自然地衔接并响应玩家的行动。

## IV. 前端开发与模块化指南

为了支撑神谕引擎的复杂逻辑，并保持代码的清晰与可维护性，我们推荐以下模块化开发策略：

1.  **创建 `engine` 目录**:
    在 `src/youth-romcom-simulator/` 下建立一个新的 `engine/` 目录，用于存放所有自动化逻辑的核心实现。
    ```
    src/youth-romcom-simulator/
    └── engine/
        ├── scheduler.ts      # 负责NPC动态日程计算
        ├── scene-engine.ts   # 负责动态场景特质激活
        ├── info-propagator.ts# 负责信息传播与流言模拟
        ├── state-manager.ts  # 负责状态清理、氛围计算等
        └── index.ts          # 组合并导出所有引擎模块
    ```

2.  **纯函数优先**:
    `engine/` 目录下的所有模块都应尽可能编写为**纯函数**。它们接收当前的世界状态作为输入，经过计算后返回一个新的状态或一个状态变更的“补丁”，而不直接修改外部变量。这使得逻辑易于测试和复用。

3.  **`变量结构.txt` 作为集成层**:
    `变量结构.txt` 的 `transform` 函数现在是所有自动化模块的“总装车间”。它负责从 `engine/index.ts` 导入各个计算模块，并按照正确的顺序依次调用它们。

4.  **UI组件可视化**:
    为新的自动化系统开发专门的调试或展示组件，例如：
    *   `GossipPanel.vue`: 一个用于查看当前`信息场`中所有流言和事实的面板。
    *   `SceneInspector.vue`: 一个显示当前场景“张力值”、“氛围”以及已激活“场景特质”的调试小部件。
    *   `NpcSchedulerDebug.vue`: 一个可以看到任意NPC当天计算出的完整行动轨迹的面板。

通过这种方式，我们既保证了神谕引擎核心逻辑的健壮性和可维护性，也为未来的功能拓展和调试提供了极大的便利。
